Contributing

Contributing to Tourmaline¶

Thank you for your interest in contributing to Tourmaline! This guide will help you get started.

Ways to Contribute¶

There are many ways to help improve Tourmaline:

üêõ Report Bugs¶

Found a bug? Please report it!

  1. Check if it's already reported in GitHub Issues
  2. If not, create a new issue with:
  3. Clear description of the bug
  4. Steps to reproduce
  5. Expected vs actual behavior
  6. Tourmaline version
  7. Operating system

Example Bug Report: 

**Bug Description:**
Division by zero doesn't raise an error in certain cases

**Steps to Reproduce:**
1. Create variable: let x = 10
2. Create variable: let y = 0
3. Calculate: let z = x / y
4. No error is raised

**Expected Behavior:**
Should raise "Division by zero" error

**Environment:**
- Tourmaline: v1.0.0
- Python: 3.9
- OS: Windows 10

üí° Suggest Features¶

Have an idea for a new feature?

  1. Check existing feature requests
  2. Open a new issue with:
  3. Clear description
  4. Use cases
  5. Example syntax (if applicable)
  6. Why it would be valuable

üìù Improve Documentation¶

Documentation improvements are always welcome:

  • Fix typos or unclear explanations
  • Add examples
  • Improve code samples
  • Translate documentation
  • Write tutorials or guides

üîß Submit Code¶

Ready to code? Great!

  1. Fork the repository
  2. Create a feature branch
  3. Make your changes
  4. Test thoroughly
  5. Submit a pull request

Code Contributions¶

Getting Started¶

  1. Fork and Clone 

    git clone https://github.com/YOUR-USERNAME/Tourmaline.git
cd Tourmaline

  2. Create a Branch 

    git checkout -b feature/your-feature-name

  3. Make Changes

  4. Write clean, readable code
  5. Follow existing code style

  6. Add comments for complex logic

  7. Test Your Changes 

    python Tourmaline.py test_file.trm

  8. Commit 

    git add .
git commit -m "Add: clear description of changes"

  9. Push and Create Pull Request 

    git push origin feature/your-feature-name

Code Style Guidelines¶

Python Code (Interpreter)¶

  • Use 4 spaces for indentation
  • Follow PEP 8 where reasonable
  • Use descriptive variable names
  • Add docstrings to functions

Example: 

def evaluate_expression(self, tokens: List[str]) -> Any:
    """
    Evaluate an expression from a list of tokens.

    Args:
        tokens: List of tokenized expression elements

    Returns:
        The evaluated result

    Raises:
        TourmalineError: If expression cannot be evaluated
    """
    # Implementation...

Tourmaline Code (Examples)¶

  • Use consistent naming (snake_case for variables)
  • Comment complex logic
  • Keep functions focused and small
  • Use meaningful variable names

Example: 

# Calculate the area of a circle
function calculate_circle_area(radius)
    let pi = 3.14159
    let area = pi * radius * radius
    return area
end

Testing¶

Before submitting:

  1. Test basic functionality
  2. Run existing example programs
  3. Test your specific changes

  4. Try edge cases

  5. Test error handling

  6. Provide invalid input
  7. Test boundary conditions

  8. Ensure errors are caught

  9. Create test files

  10. Add .trm files demonstrating your feature
  11. Include both success and error cases

Pull Request Guidelines¶

Title Format: - Add: Feature description for new features - Fix: Bug description for bug fixes - Docs: Documentation improvement for docs - Refactor: Code improvement for refactoring

Description Should Include: - What changes were made - Why they were necessary - How to test them - Any breaking changes - Related issues (if any)

Example PR Description: 

## Changes
Added support for list slicing with `list[start:end]` syntax

## Motivation
Users requested the ability to extract sublists more easily

## Testing
- Added test_slice.trm with various slice examples
- Tested with positive and negative indices
- Tested edge cases (empty lists, out of bounds)

## Breaking Changes
None

## Related Issues
Closes #42

Feature Development Process¶

1. Planning¶

  • Discuss the feature in an issue first
  • Get feedback from maintainers
  • Consider backward compatibility
  • Think about edge cases

2. Implementation¶

  • Start with tests/examples
  • Implement incrementally
  • Keep commits focused
  • Write clear commit messages

3. Documentation¶

  • Update relevant docs
  • Add examples
  • Document new syntax
  • Update README if needed

4. Review¶

  • Self-review your code
  • Test thoroughly
  • Ensure it works on different systems
  • Request feedback
